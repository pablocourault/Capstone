PROCESOS

- Navegar el sitio, ver el hotel (no requiere nada)
- Registrarse
- Login / Logout
- Consultar habitaciones disponibles por rango de fechas (modelos: ROOM, BOOKINGS)
- Consultar habitaciones disponibles y hacer reserva (modelos: ROOM, BOOKINGS)
* Enviar y recibir mensajes de la recepción tipo DM (hacerlo si no se complica, modelo: MESSAGES)
* Dejar comentarios sobre el servicio del hotel al hacer el Check Out (modelo: COMMENTS)
* Pedir servicios para una habitación (modelos: SERVICES, CONSUMPTIONS)
  (el usuario recibe un mensaje en su casilla luego de hacer el pedido).
* Obtener gastos a pagar (modelo: CONSUMPTIONS)
* Check In: agregar usuario a GUESTS, agregar amount de BOOKINGS en consumos y
            cambia el estado de la reserva a checkin TRUE.
            El usuario solo puede tener una sola reserva como Check-In por vez.
* Check Out: habilitado con código de reserva, eliminar usuario de GUESTS, 
             guardar comentario (si lo desea), validará el código de booking (para demo de 'ABC123').
             borra la reserva correspondiente y todos los consumos asociados a la misma.
             Osea que borra al usuario de GUESTS, a la reserva de BOOKINGS y a todos los consumos de CONSUMPTIONS.
- Borrar reserva: El Delete de la reserva solo lo puede hacer si no hizo el Check-In.
                  Si el Check-In es False aparecen las opciones Delete y Check-In.
                  Si el Check-In es True aparece sólo la opción Check-Out.

MODELOS

* USERS

* GUESTS
  - guest (usuario, si está en este modelo, 
           es porque hizo el check-in -se lo agrega en ese momento-, lo habilita a hacer consumos,
           se borra al hacer el check-out)
  - booking (tiene una instancia por cada reserva, se borra una instancia por cada check-out).


* ROOM
 - description
 - room-type
 - quantity
 - rate


* BOOKINGS (es la reseva, no importa las habitaciones, está bien pensado asi)
 - user
 - singles
 - doubles
 - triples
 - quadruples
 - checkin_date
 - checkout_date
 - checkout_code (código necesario para hacer el ckeck-out y que cancela los consumos
                  hipotéticamente se lo entregarían al huésped al momento de hacer el pago)
                  (la disponibilidad de reserva se hace contando la cantidad reservadas
                   para esa fecha por room-type, menos la quantity que hay en ROOM
                   por room-type.)
 - amount       (al hacer la reserva, se le guarda el monto total por las habitaciones, es decir:
                 se le "congela" la tarifa porque después puede cambiar,
                 OJO, cuando hace el check-in se le inicializan los consumos con este monto.
                 Este campo se utiliza para eso y para ir acumulando el total, cuando se van 
                 incluyendo consumos.)
 - checkin       Boolean, True when Guest did check in.


* CONSUMPTIONS
 - user
 - booking
 - description
 - date
 - amount
   (El usuario cancelaría todos los consumos correspondientes a un booking.
    al momento de hacer el check-out, con el código de pago que le suministraría
    el administrador o Front End; el cual es el campo checkout_code.
    O sea que cuando hace el check-out de una reserva, hay que borrar todos los consumos.
    Para los propósitos de uso DEMO el checkout_code podrá ser tambén 'ABC123'.
    El monto se va sumando al amount del booking para que aparezca el total completo,
    en la página mybookings.html y se pueda ver al momento de hacer el check-out.)

* SERVICES
 - description
 - message (el mensaje que recibe el usuario, luego de hacer el pedido del servicio)
 - rate

* MESSAGES (HACER LO DE MENSAJES SI NO SE COMPLICA)
 - user
 - date
 - Destinatario (para el usuario el destinatario siempre es la administración del hotel)
 - message
 - state (leído / no leído)

* COMMENTS
 - user
 - date
 - comment
 - score
